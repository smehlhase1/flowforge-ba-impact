FlowForge · Partner Integration — PROD-13946

Onboard partners without engineering

Adding a new partner to the CIL platform today requires an engineer to hand-author Go constants, YAML stanzas, rate limiter entries across 5 files, auth manifests, and adapter skeletons — all from tribal knowledge spread across Confluence, prior PRs, and institutional memory. Integration managers cannot initiate or track an onboarding without engineering at every step. /flowforge.integration changes that: plain-language prompts, 9 guided phases, 5 ready-to-merge artifacts, and automatic Jira tickets — no Go knowledge required.

The problem

Onboarding a partner today requires an engineer at every step

The CIL platform has 5 onboarding steps. Today an engineer must be involved in all 5 — not because the decisions are complex, but because the tooling is entirely engineering-facing. An integration manager who knows the partner, the endpoints, and the LoB cannot self-serve any of it.

What a CIL onboarding actually involves: For a partner like RBB or DKB, the integration team needs: (1) a Go ProductCode constant and &Product{...} struct in products.go, (2) an access control YAML stanza listing every permitted endpoint, (3) a rate limiter entry copied to 5 environment config files, (4) an auth client manifest for the Vault secret, and (5) a Go adapter skeleton at the correct package path. Every one of these requires an engineer to know the naming conventions, find the right file, derive the correct enum values, and not make a uniqueness mistake. None of it is conceptually hard — it is all pattern-following from tribal knowledge.
The integration manager's current workflow: An integration manager receives a partner onboarding request and immediately has to loop in engineering. They write a Jira ticket describing what the partner needs. An engineer interprets that ticket, looks up a prior onboarding PR (e.g. RBB CIL-6247 or DKB CIL-6516), manually copies and adapts the artifacts, creates more Jira tickets, and hands back. The integration manager has no visibility into which step is complete, which file has been edited, or what is blocked — unless they ask. There is no dashboard, no checkpoint, and no way to self-serve any part of the flow.

The command

/flowforge.integration — 9 guided phases

Starts with "What is the name of the business partner?" — no arguments, no technical prerequisites. Every identifier is derived automatically. The command pauses at the Admin UI checkpoint (a real-world dependency) and resumes when the partner ID is in hand. All 5 engineering artifacts are generated in Phase 8 and Jira tickets are created in Phase 9.

Companion command

/flowforge.integration.status — onboarding dashboard

Read-only. Reads products.go, all checkpoint files, and open Jira tickets labelled partner-onboarding at runtime. Shows ONBOARDED partners, IN PROGRESS onboardings (with phase + timestamp), and JIRA OPEN ITEMS. Supports --partner, --status, and --country filters. Makes no writes of any kind.

Why live reads matter

Every data source read at runtime — no stale copies

The pattern that makes this safe: every source of truth in the codebase is read at command runtime, not at skill-write time. The skill stays current as endpoints, enum values, and product codes change without requiring skill file edits.

The live-read cascade
P2
LoB list → simplesurance-api.yml tags at runtime
Never hardcoded. When a new LoB is added to the API (e.g. a new tag in the OpenAPI spec), it automatically appears in the integration manager's selection list. No skill file edit required.
P2
InsuranceType enum → rtqpolicy.proto at runtime
The INSURANCE_TYPE_* enum value is read directly from the proto source. An integration manager never needs to look up an enum name — the mapping is derived automatically and will stay correct as new insurance types are added to the proto.
P3
Endpoint list → simplesurance-api.yml operationIds at runtime
Filtered to the selected LoB tag. New endpoints added to the API spec appear immediately in the next onboarding run. Roles and access control resources are derived from this live list — no manual mapping table.
P4
Product code uniqueness → products.go at runtime
products.go is the source of truth — no separate registry file that could go stale or be missed in a PR. The skill greps all existing "PC[^"]*" string values and rejects any collision immediately. Confirmed codes are reserved for the session to prevent double-use within a single run covering multiple countries.
P8
Auth schema → clientmgr.go and client.go at runtime
The auth client manifest template fields are derived from go/code/auth/internal/clientmgr/clientmgr.go and go/code/auth/store/client.go. If the auth client schema changes, the manifest template reflects it on the next run.
Where this won't land as predicted
constraint
Admin UI gate is unchanged
The selling partner account must still be created manually in Admin UI by the Luis/Integration team before artifacts can be completed. This is a real-world external dependency — the command accommodates it with a pause-and-resume checkpoint but cannot eliminate the wait. Onboarding calendar time is therefore not reduced by the same factor as active effort time.
constraint
New flow handlers still require engineering
When a partner uses an insurance backend not already wired into CIL (Phase 6 → new flow handler), the adapter skeleton is a starting point — an engineer must still implement the actual flow logic. The command generates the skeleton and creates the 3-ticket breakdown, but does not implement the backend integration. This is correct scope: the command covers onboarding to existing endpoints, not new development.
soft risk
Auth manifest credentials are always [MANUAL]
The auth client manifest template intentionally leaves secret, client_auth_userid, and client_auth_user_type as [MANUAL]. Real credentials are distributed via secrets management and must be filled in by the team before deployment. This is a security constraint, not a gap — but it means the generated manifest is not deployable as-is.
soft risk
Additions mode scope
When an already-onboarded partner adds a new country or LoB (additions mode, T007), the command generates only the delta artifacts — new product code(s) and additional access control and catalog stanza entries. It does not re-generate the auth manifest or adapter skeleton, which already exist. Integration managers must know to use additions mode rather than re-running from scratch.
soft risk
Jira MCP availability
Ticket creation in Phase 9 uses the Atlassian MCP. If the MCP is unavailable, the command falls back to printing ticket templates for manual creation. All artifact generation is unaffected — only the Jira step degrades gracefully.

Build sequence

8 tasks — three delivery groups

Tasks are structured so each group is independently releasable. The guided input phases ship first (the foundation); artifact generation and Jira output ship second; resume mode and the status dashboard ship third and are immediately usable alongside the main command.

Group 1 — Guided input
Data collection foundation
CIL-6785 — T001: Partner identity, LoB, endpoints
CIL-6786 — T002: Product code generation + validation
CIL-6787 — T003: Admin UI checkpoint, flow handler, backend IDs
These three tasks implement Phases 1–7. All subsequent phases depend on the slug, LoB, InsuranceType, selected endpoints, partner ID, and backend IDs produced here. The live reads from simplesurance-api.yml and rtqpolicy.proto are established in T001. The pause-and-resume checkpoint pattern is established in T003.
Group 2 — Artifact generation + Jira
Deliverable production
CIL-6788 — T004: Catalog, access control, rate limiter
CIL-6789 — T005: Auth manifest template + adapter skeleton
CIL-6790 — T006: Jira ticket creation output
Implements Phase 8 (5 artifacts) and Phase 9 (Jira output). Depends on all Group 1 outputs. T004 covers the three config artifacts (CIL steps 2 and 3). T005 covers the two implementation artifacts (CIL steps 1 and 4). T006 closes the loop with tracked work items in Jira — 1 ticket for flow-handler-reuse, 3 tickets for new.
Group 3 — Resilience + visibility
Resume mode + status dashboard
CIL-6791 — T007: Resume + additions mode
CIL-6792 — T008: /flowforge.integration.status dashboard
T007 enables checkpoint persistence — critical because the Admin UI gate may take minutes to days. Without resume, re-running would require re-entering all prior inputs. Additions mode handles the common case of expanding an existing partner to a new country or LoB. T008 is independently implementable and provides the live onboarding dashboard that answers "where are we with partner X?" without engineering involvement.
